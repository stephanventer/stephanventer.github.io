Welcome. 60 minutes. We all use Devin daily so I'll skip the basics. Focus today: the workflow that wraps the AI — where the human gates are, what good looks like at each stage, and the anti-patterns to avoid. Worked example in the middle. Cheat sheet at the end you can screenshot.
Why this exists

Small. Reviewable.
Shippable. Owned.

We're fast because we plan well, not because we skip planning. The agent writes the code — you own the outcome.

The point of this whole process is to make work small, reviewable, and shippable — and to keep ownership human even when the typing is automated. If you take one thing from today: AI accelerates the parts after the plan. It doesn't replace the plan.
00The cycle at a glance

Six phases. One feedback loop.

Six phases. Phase 02 is optional — skip it for small SUs that don't need cycle-level architecture work. Everything else is non-negotiable. Note the feedback loop: each PR feeds back into the next SU plan. Lessons travel.

Investigate
Before you plan,
learn.
Phase 01. The temptation is always to skip this and jump to planning. Don't. The investigation phase is where you convert gut feel into evidence and avoid building the wrong thing twice.
01Investigate

What to research

Sources

  • Current architecture — code paths, services, contracts
  • Usage data — BigQuery, Google Analytics
  • Prior work — Jira, Confluence, past PRs
  • Spikes — small prototypes when a question can't be answered by reading

Where it lives

Confluence: Project > Engineering > Research > [Topic]

Link the research doc from the SU plan. Don't bury it in Slack.

⚑ Convert gut feel into evidence.
✓ DONE WHEN
Research doc exists in Confluence · key unknowns listed explicitly · decisions backed by either data or "we accept this risk because…"
Four sources. BigQuery and GA are underused — check actual usage before you assume what users do. Jira history is gold for "have we tried this before and why didn't it stick?" Prototypes only when reading the code can't answer the question. DoD: a doc, a list of unknowns, and named decisions. No "TBD."

02Cycle plan

What to cover

Architecture Database UI / UX Security Performance Testing Observability Migration / rollout

Output

  • A picture paints a thousand words — diagram before you describe
  • Decisions named, alternatives mentioned
  • Leads in the loop early

Not the goal

  • Solving every unknown
  • Pixel-perfect design
  • Full task breakdown — that's 03
✓ DONE WHEN
Key cycle-level decisions documented · diagrams in place · open questions named (not silenced)
Cycle planning is about shape, not slices. Architecture, DB design, UI flow, security boundary, perf budget, test strategy. Output is diagrams plus decisions — not prose. Avoid the trap of trying to solve everything here — task-level breakdown belongs to the SU plan.

SU plan
Plan the slice

Phase 03 is where most of the time and most of the value lives. A great SU plan makes implementation almost mechanical. A bad SU plan turns implementation into a series of guesses by the agent.
03Anatomy of the prompt

What the agent needs from you

## context Add "Recently Viewed Jobs" so people can resume work fast. ## links - Brief: confluence.tradify/jobs/recent-viewed-brief - Func spec: confluence.tradify/jobs/recent-viewed-spec - Design: figma.com/file/... - Research: confluence.tradify/research/job-resume-patterns ## unknowns - Should "viewed" mean opened, or interacted with? - Retention window — 30 days? Forever? - Should sync across devices or stay local? ## constraints - Must work offline - No new backend endpoints if avoidable ## ALWAYS ASK CLARIFYING QUESTIONS FIRST. Do not propose a plan until I confirm.

The five blocks

  • Context — what & why, one paragraph
  • Links — brief, spec, design, research
  • Unknowns — the things you're not sure about
  • Constraints — what bounds the solution space
  • The instruction — clarify before planning
⚑ Always ask for clarifying questions first.
Five blocks. The last one is non-negotiable. If you skip "ask clarifying questions first," the agent will fill the gaps with assumptions. The unknowns block is where you list everything you'd otherwise have to guess.
03Task breakdown

The rules

Required

  • One ticket = one PR. No exceptions.
  • Dependency graph — explicit, drawn
  • Each task independently testable

Smells

  • "Implement feature X" — too big
  • "Refactor and add Y" — that's two PRs
  • Tasks that touch unrelated parts of the codebase
⚑ If a task or PR can't be reviewed in less than 30 minutes, split it.
One ticket = one PR. Hammer this. The dependency graph isn't a nice-to-have — it's how the team understands what blocks what, and it's how the implementation phase can parallelize. The 30-minute rule is the simplest test.
03The planning loop

Iterate until the plan reads boring

The loop. Send prompt, get clarifying questions, answer them, get a plan, review, iterate. "Boring" is a compliment here — when the plan stops surprising you, it's ready. Then create the empty Confluence page yourself (so you control the URL and breadcrumb), and have Devin populate it.
03Where the plan lives

Confluence is the source of truth

Path: Project > Engineering > SU01 — [Feature name]

Two ways to make changes

Edit by hand

Open the page and edit it yourself. Good for small tweaks, typo fixes, or quick clarifications.

Ask Devin

Ask Devin to read the page, apply the changes, and update it. Good for bigger reshapes or new sections.

Either works — pick whichever is faster for the change you need.

Confluence is the home for the plan. Two equally valid ways to make changes. The only thing to avoid is having two people editing at once without coordinating — same as any shared doc.
03SU plan

Plan is done when…

  • Plan lives in Confluence under Project > Engineering
  • Dependency graph present (and accurate)
  • Every task is reviewable in <30 min
  • Lead and EM have reviewed
  • Zero "TBD" sections
Five exit criteria. The "TBD" one matters most — TBDs in plans almost always become assumptions in code. If you have TBDs, the plan isn't done.
EXWorked example · the prompt

SU01 — Recently Viewed Jobs

A real-shaped (not real) SU we'll walk through end to end.

## context People open the app to resume the job they were on. Currently they have to search for it again every time. Add a "Recently Viewed Jobs" section to the home screen showing the last 5 jobs they opened. ## links - Brief: confluence.tradify/jobs/recent-viewed-brief - Func spec: confluence.tradify/jobs/recent-viewed-spec - Design: figma.com/file/recent-jobs-v3 - Research: confluence.tradify/research/job-resume-patterns (BigQuery: 73% of sessions reopen a recent job) ## unknowns - "Viewed" definition — first open, or any interaction? - Retention — 30 days? Forever-til-replaced? - Sync across devices, or local-only? - Behaviour when offline and list is empty? ## constraints - Must work offline - No new backend endpoints unless absolutely required - Telemetry: existing job_opened event is enough, add a section_tapped event ## ALWAYS ASK CLARIFYING QUESTIONS FIRST. Do not propose a plan or write code until I confirm answers.
A realistic SU prompt. Notice: context is one paragraph. Links are named. Unknowns are real questions. Constraints bound the solution space without prescribing the solution. The final two lines are the gate.
EXWorked example · the breakdown

Task graph (after clarification)

7 tasks · 7 PRs · parallelizable from T2 onwards · each reviewable in <30 min

Same SU broken into tasks. Each node is one PR. The graph shows what can run in parallel — once T2 lands, T3 and T5 can go at the same time. T4 then unblocks T6 and T7. You can't make these tradeoffs without a graph.

Phase 04. Short phase. Mechanical. But still has a gotcha — the back-link.
04Tickets

The handover

What you give Devin

  • Confluence page URL
  • Sprint number
  • Epic name
  • Project key if not default

What Devin does

  • Creates one ticket per task
  • Sets sprint, Epic, project
  • Links each ticket back to its task on the Confluence page (bidirectional)
⚑ New Devin session for tickets. Don't reuse the planning session — context bleed.
✓ DONE WHEN
Every task has a ticket · tickets live in the right Epic + sprint · Confluence page shows ticket links next to each task
New Devin session. Always. The planning session has too much context and Devin will second-guess the plan instead of just making tickets. The back-link to the Confluence page matters — without it, the plan and the tickets drift.
05
Implement
Plan first.
Code second.
Verify always.
Phase 05. The fun part. Also the part where people skip the planning gate because "it's just a small task." Don't.
05Implement

Per-task workflow

Setup

  • Your tool of choice — Claude Code, GitHub Copilot, Devin, local
  • Atlassian MCP connected — agent reads the ticket directly
  • Feed the ticket as context. Add anything else useful.

Workflow

  1. Plan first. Always.
  2. Ask for clarifying questions. Avoid assumptions before code.
  3. Confirm plan, then implement
  4. Review per-step or as-whole — your call
  5. Run and verify yourself. You own the code.
⚑ The agent typed it. You shipped it.
Same rules as SU planning, just at task level. Plan first, clarify first. Review style is personal. What's non-negotiable: you run the thing yourself before it goes anywhere.
06
PR
Draft until proud.
Ready when reviewable.
Phase 06. Short but with two explicit signals — Draft vs Ready — that we want everyone using consistently.
06PR

Two states. One signal each.

⊘ Draft

Work in progress. Do not request review. CI runs, you iterate, no one is on the hook for reading it yet.

▸ Ready

Reviewable and mergeable. You've ticked it as owner. CI green. Testing notes attached.

After PR exists

  • Hand the PR to Devin → ask for comprehensive testing notes
  • Owner tick (you) → flip from Draft to Ready
  • Request reviewers
✓ DONE WHEN
PR Ready · CI green · testing notes in description · owner tick · linked to ticket
Draft means WIP — don't ping people. Ready means it's truly ready. Devin generates the testing notes after the diff is stable.

Anti-patterns

What not to do

✕ Skipping clarifying questions. The agent fills gaps with assumptions you'll only find in code review.
✕ Monolithic PRs. "One ticket = one PR" violated. Reviewers tap out. Bugs hide.
✕ Plans without dependency graphs. If you can't draw it, you don't understand the work.
✕ Vague prompts. "Devin will figure it out" → Devin invents requirements. Garbage in, plausible-looking garbage out.
✕ Skipping the review gate to look fast. You shave a day. You add a week of rework. Always.
✕ Treating Ready like Draft. "It's basically done" — reviewers waste time on WIP code.
✕ Not running it yourself. The agent shipped. You owned. Always verify before Ready.
✕ TBDs in the final plan. TBDs always become assumptions in code. Resolve before tickets.
Eight anti-patterns. Worth reading aloud. The pattern across all of them: cutting the human gates to look fast.
